# ticket-incubate — 아이디어를 티켓으로 부화

사용: `/ticket-incubate [아이디어 한 줄]` 또는 그냥 대화 중 발동.

이슈가 없는 막연한 아이디어를, 착수 가능한 티켓 하나로 익혀 GitHub 이슈로
부화시킨다. 워크벤치 수명주기에서 `/task-start` 바로 앞 단계다.

핵심 비유는 **인큐베이션**이다 — 알을 깨자마자 모양을 강요하지 않는다. 먼저
따뜻하게 품어 키우고(발산), 충분히 자랐을 때 부족한 곳만 손보고(인터뷰),
다 차면 부화시킨다(이슈 생성). 순서를 뒤집으면 — 처음부터 4요소를 캐물으면 —
아직 모양이 안 잡힌 아이디어를 틀에 우겨넣어 죽인다.

## 종착점: 티켓 4요소

이슈 본문은 `.github/ISSUE_TEMPLATE/task.md`의 4요소 형식을 따른다:

- **배경** — 왜 지금 이게 필요한가. 무엇이 불편하거나 비어 있나.
- **목표** — 이 티켓이 끝나면 무엇이 달라지나. 한 문장으로 말할 수 있어야 한다.
- **완료 기준** — 무엇을 보고 "끝났다"고 판단하나. 검증 가능한 체크 항목.
- **이번에 하지 않을 것** — 가까이 있지만 이 티켓에선 손대지 않을 것. 범위를
  좁혀 티켓이 커지는 걸 막는다.

이 네 가지가 다 차는 것이 인터뷰의 종착점이다. 그 전엔 발동하지 않는다.

## 흐름

### 1. 발산 — 같이 굴린다

사용자가 아이디어를 꺼내면 맞장구치며 같이 키운다. 연관된 가능성, 비슷한
사례, 자연스러운 확장을 제안해 아이디어를 입체적으로 만든다.

이 단계에서 4요소를 캐묻지 않는다. 아직 발산 중인 아이디어에 "완료 기준이
뭐죠?"를 들이대면 사고가 닫힌다. 사용자가 충분히 말하게 둔다.

### 2. 전환 신호 — 언제 인터뷰로 넘어가나

다음 중 하나가 보이면 결손 인터뷰로 전환한다:

- 대화만으로 **4요소 중 2개 이상을 초안으로 적을 수 있게** 되었다, 또는
- 사용자가 **수렴 신호**를 보낸다 ("정리하자", "만들자", "티켓으로", "이슈로").

전환 전에는 발산을, 전환 후에는 수렴을 한다. 이 신호가 둘 사이의 경계다.

### 3. 가드 — 이미 정했나 / 망했나

인터뷰로 넘어가기 전에 `/docs-query`를 호출해 이 아이디어와 관련된 선행
지식을 확인한다. 기존 결정(decisions)이 방향을 정해뒀거나, 기존 교훈
(lessons)이 "이 접근은 망했다"고 말할 수 있다.

- **선행 결정 히트** → 아이디어를 그 결정에 맞춰 조정하거나, 뒤집겠다면 그
  근거를 티켓 배경에 명시한다.
- **선행 교훈 히트(아이디어를 죽임)** → 사용자에게 표면화한다. 이미 망한
  접근이라면 **티켓을 만들지 않고 종료하는 것도 정상**이다. 사용자가 그래도
  다른 각도로 가겠다면 그 각도를 티켓에 담는다.

조용히 진행하지 않는다 — 가드의 목적은 같은 실수·중복 결정을 막는 것이다.

### 4. 결손 기반 인터뷰 — 빈 곳만, 하나씩

지금까지의 대화를 4요소에 매핑한다. 그리고 **비어 있거나 약한 항목만** 묻는다.

- 이미 대화에서 나온 요소는 다시 묻지 않는다 — 사용자는 같은 말을 반복하기
  싫어한다. 채워진 것은 "배경은 …로 이해했어요"처럼 확인만 한다.
- 한 번에 한 질문씩. 4요소를 한꺼번에 나열해 던지지 않는다.
- **home**(이슈가 살 곳: workbench인지 특정 codebase인지)도 이때 확정한다.
  codebase 이슈면 그 repo에 자체 이슈 템플릿이 있는지 확인하고, 있으면 그걸
  우선한다(repo 컨벤션 우선).

질문은 결손을 메우는 최소한으로 한다. 4요소가 다 차면 멈춘다.

### 5. 티켓 우선 종결 — 조사가 남아도 먼저 만든다

4요소가 채워지면, 설령 깊은 조사·탐색이 남아 있더라도 **티켓 생성을 먼저
권한다.** 더 파고들자고 대화를 끌지 않는다.

이유: **대화는 휘발이다.** 티켓이 생기기 전에 발견한 것은 이 대화가 끝나면
사라진다. 반면 티켓 이후의 조사는 task 안에서 `log`·harvest로 축적된다.
그러니 "조사는 task 안에서 계속하면 된다 — 일단 티켓부터 만들자"가 옳다.

### 6. 승인 게이트 — 보여주고 받는다

생성 직전, 제목과 본문 4요소 전체를 사용자에게 보여주고 **명시적 승인**을
받는다. 이슈 생성은 외부에 공개되는 행위다 — 묻지 않고 만들지 않는다.

이때 **priority 라벨을 함께 제안**한다 (`priority:P0`~`P3`, 추천 사유 한 줄).
priority는 "지금 다른 모든 이슈 대비 얼마나 급한가"라는 cross-issue 판단이라
에이전트가 자동 부여하면 거의 틀린다 — 제안만 하고 **사람이 확정·수정**한다.
승인 시 `gh issue create --label priority:P?`로 함께 단다.

type 라벨(`enhancement` 등)·area 라벨은 굳이 제안하지 않는다 — 이 워크벤치의
작업은 거의 전부 enhancement라 변별력이 없고, area 분류는 아직 소비자가 없다
(필요해지면 그때 사용 양상에서 만든다). 사용자가 직접 요청하면 단다.

### 7. 생성

승인되면 `gh issue create --title <제목> --body-file <본문>` 으로 만든다.
codebase 이슈면 해당 repo 디렉토리에서 실행한다(`gh -R <owner/repo>` 또는
해당 repo에서). 생성 후:

- 이슈 URL을 보고한다.
- `/task-start <ID>`로 바로 착수할지 제안한다.

## 아이디어가 티켓 하나보다 클 때

인터뷰 중 아이디어가 여러 repo에 걸치거나, 독립적으로 완료 가능한 덩어리가
여럿이라는 게 드러날 수 있다. 그래도 한 호출 = 한 티켓을 지킨다.

- **가장 먼저 할 첫 티켓 하나**를 골라 4요소를 갖춰 완성하는 게 우선이다.
- 나머지 덩어리는:
  - **조각이 명확하면**(각각 4요소를 바로 채울 수 있으면) 후속 티켓을 추가로
    개설할 수 있다.
  - **불명확하면** 첫 티켓에 `## 남은 조각` 마커로 댓글을 달아 맥락을
    보관하고, 다음에 `/ticket-incubate`로 그 조각을 다시 아이디에이션하라고
    추천한다. 이 마커는 다음 호출이 읽을 진입점이 된다.

큰 아이디어를 한 번에 다 분해하려 들지 않는다. 첫 조각을 잘 벼리는 게 낫다.

## 엣지 케이스

- **`gh` 인증 실패 / 대상 repo의 issues 비활성** → 4요소 본문을 대화에 그대로
  출력하고, 사용자가 웹에서 직접 만들도록 안내한다. 본문을 잃지 않게 한다.
- **가드가 아이디어를 죽임** → 티켓 없이 종료하는 것이 정상 종료다. 죽은
  이유(어떤 선행 교훈·결정)를 사용자에게 남긴다.
- **중간 이탈** → 아이디어 개진은 휘발이라 정리할 것이 없다. 다음에 다시
  `/ticket-incubate`로 시작하면 된다.

## 경계 — brainstorming과 다르다

`superpowers:brainstorming`은 **지금 구현할 것**의 설계·스펙을 짠다.
ticket-incubate은 **나중에 할 일**을 잘 벼린 티켓으로 포착한다. 사용자가
"이거 만들자, 바로 짜자"면 brainstorming, "이거 해야 하는데 / 언젠가 / 티켓
만들어두자"면 ticket-incubate이다.
